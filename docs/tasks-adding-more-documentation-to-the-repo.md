## Task 1: Create ARCHITECTURE.md

Write comprehensive system architecture documentation explaining how the financial-analyzer application is structured. Include directory layout with descriptions of each major folder (app/, lib/, scripts/, __tests__/), a request flow diagram showing how user interactions reach the database and back, component hierarchy showing React component relationships, and key design decisions with rationale (why SQLite, why Next.js App Router, why the chosen tech stack). Add ASCII diagrams for visual clarity. Target length: 1,500 words. This serves as the primary onboarding document for new developers to mentally map the codebase in under 30 minutes.

## Task 2: Create API.md with complete endpoint documentation

Document all REST API endpoints in the application with comprehensive details for each. For every endpoint under app/api/, provide: HTTP method and path, description of purpose, all parameters (path, query, body) with types and constraints, example request with curl command, example response with full JSON schema, error codes and their meanings, caching behavior, and any special notes. Cover these endpoints: GET /api/market/chart/[ticker], GET /api/market/options-overlay, GET /api/options/snapshot, GET /api/options/projection, GET /api/options/ai-forecast, GET /api/fear-greed, GET /api/reports, GET /api/reports/latest, GET /api/reports/[date], and POST /api/reports/generate. Target length: 2,000 words. Include a table of contents at the top for easy navigation. Ensure all code examples are copy-pasteable and tested.

## Task 3: Create CONTRIBUTING.md

Write a comprehensive contribution guide that covers the entire developer workflow from initial setup through PR merge. Include: development environment setup (expand from DEV.md), worktree workflow with practical examples, code standards for TypeScript (strict mode, no any, return type annotations), React component conventions (functional components, CSS modules, prop interfaces), API route patterns (input validation, error shapes, cache headers), testing requirements (unit tests for logic, integration tests for APIs, E2E for critical flows, 80% coverage minimum), PR checklist (tests pass, linting clean, docs updated, no console errors), detailed PR review process, and step-by-step walkthroughs for common tasks like adding a new API endpoint or React component. Target length: 1,200 words. Make it actionable enough that a first-time contributor can submit a clean PR without asking questions.

## Task 4: Create DATABASE.md

Document the complete database schema, operations, and maintenance procedures. Cover SQLite setup and database location (data/reports.db), current schema version (v4), full table definitions for reports and option_prices tables with column descriptions and data types, all indexes with explanations of why they exist, unique constraints, common SQL query examples (selecting recent option prices, listing available contracts, getting latest report), backfill script documentation with usage examples, backup procedures (manual and automated via systemd timer if applicable), recovery procedures with step-by-step commands, database inspection techniques using sqlite3 CLI, and performance optimization tips (using indexed columns, LIMIT clauses, quarterly VACUUM). Target length: 1,500 words. Include practical examples that operators can copy-paste for common maintenance tasks.

## Task 5: Create TROUBLESHOOTING.md

Build a comprehensive self-service debugging guide covering the most common issues developers and operators encounter. For each issue provide: symptom description (exact error messages, when it occurs), root causes (list all possible reasons), solutions ordered from quickest to most thorough (quick fix, thorough fix, nuclear option), verification steps to confirm the fix worked, and prevention tips. Cover these issues: "Module not found" errors after npm install, port already in use errors, database file not found, API returning 404 or no data, React components not rendering, TypeScript compilation errors, E2E tests timing out or failing, build failures, systemd service won't start or crashes, and data inconsistencies in the database. Target length: 1,500 words. Organize with clear headings so developers can quickly scan for their specific error message. Include cross-references to related documentation sections.

## Task 6: Create DEPLOYMENT.md

Document the complete production deployment architecture and procedures. Cover: environment overview table (production vs dev worktrees with paths, ports, branches, who manages each), detailed deployment flow from code push through production restart (GitHub Actions trigger → webhook → deploy.sh → build → systemd restart), manual deployment procedure for emergencies when CI/CD is unavailable, pre-deployment checklist (tests pass, PR approved, CI green, migrations tested, backup taken), rollback procedure with specific git commands to revert to last working commit, monitoring commands (systemctl status, journalctl for logs, health check endpoints), and handling database migrations during deploy. Target length: 1,200 words. Make procedures specific enough that an operator can execute without developer assistance, and ensure rollback can complete in under 5 minutes.

## Task 7: Create TESTING.md

Document the complete testing strategy across all three testing layers. Explain the three-layer approach: unit tests with Vitest for pure functions and business logic in __tests__/unit/, integration tests with Vitest for API routes and database interactions in __tests__/unit/api/, and E2E tests with Playwright for full user workflows in e2e/. For each layer provide: scope definition, file organization conventions, multiple working code examples showing real test patterns, commands to run tests locally, and coverage targets. Include common testing patterns: mocking API responses with vi.mock(), testing error cases with expect().rejects.toThrow(), testing async operations, testing React components if applicable. Document CI/CD integration showing how tests gate PR merges. Target length: 1,500 words. Make examples detailed enough that new developers can copy the pattern for their own tests.

## Task 8: Create TYPESCRIPT.md

Document TypeScript configuration, conventions, and common patterns used in the codebase. Explain tsconfig.json settings with inline comments (strict mode, module resolution, path aliases like @/*), core conventions (interfaces for public APIs vs types for internal data, never use any - use unknown with type guards instead, always specify function return types, prefer const over let), common patterns (API response wrapper types, database row interfaces, component props interfaces), IDE setup recommendations for VSCode with required extensions and settings.json configuration, and a debugging section for common TypeScript errors with solutions ("Property X does not exist on type Y", "Type X is not assignable to type Y"). Target length: 800 words. Include side-by-side good vs bad examples to make conventions clear.

## Task 9: Create DOCS.md navigation index

Create a central documentation index that serves as the entry point for all documentation, organized by reader persona. Provide clear reading paths for: new developers (README → ARCHITECTURE → DEV → CONTRIBUTING → TESTING), API consumers (README → API → TROUBLESHOOTING), operators/DevOps (DEPLOYMENT → TROUBLESHOOTING → DATABASE), code reviewers (CONTRIBUTING → TESTING → ARCHITECTURE), and feature contributors (CONTRIBUTING → TESTING → API). Include a complete table listing all documentation files with one-sentence purpose descriptions. Add quick links section for frequently accessed resources (GitHub issues, CI/CD pipeline, production dashboard). Target length: 500 words. Use clear visual hierarchy with emoji or other markers to make navigation intuitive at a glance.

## Task 10: Update README.md

Expand the current basic Next.js boilerplate README into a proper project overview and quickstart guide. Add: brief project description (1-2 sentences about what financial-analyzer does), clear quickstart section (clone, install, run dev server in 5 steps or less), features overview as bullet list (SPX charts, option price overlay, reports generation, watchlist), prominent "📚 Documentation" section pointing to DOCS.md as the main entry point, technology stack overview (Next.js 14, React, TypeScript, SQLite, Tailwind CSS), link to CONTRIBUTING.md for developers wanting to contribute, and link to DEPLOYMENT.md for production setup. Keep it concise (under 400 words) since detailed docs are in other files - README should be the 2-minute overview that tells developers if they're in the right place and where to go next.

## Task 11: Expand DEV.md

Enhance the existing development workflow documentation with more detailed guidance. Add to current content: more detailed npm script explanations (what each script does, when to use it), worktree best practices (port allocation, branch naming, when to create vs when to work in main repo), common development tasks section with step-by-step guides (adding a new API endpoint with file structure and boilerplate, adding a React component with props and styling, modifying database schema with migration steps), debugging tips (using Chrome DevTools, inspecting SQLite database, reading Next.js error messages), and environment variable configuration (.env.local setup, available vars). Target additions: 500 words on top of existing content. Focus on practical "how do I..." scenarios that new developers encounter in first week.

## Task 12: Add JSDoc comments to complex functions

Add comprehensive JSDoc documentation to all complex utility functions, API route handlers, custom React hooks, database query functions, and business logic in lib/. For each function include: one-line description, @param tags for all parameters with types and descriptions, @returns tag with return type and description, @throws tag if function can throw errors, and usage example in code comment when pattern is non-obvious. Prioritize these areas: lib/ directory utilities (calculations, data transformations), app/api/ route handlers (request/response contracts), any Black-Scholes or Greeks calculations if present, database query builders, custom React hooks if any. Follow JSDoc standard format. Estimated scope: 30-50 functions. Use TypeScript types in JSDoc (@param {number} not just "number"). Make comments detailed enough that IDE autocomplete provides useful guidance without reading source code.

## Task 13: Create documentation validation script

Write a shell script (scripts/validate-docs.sh) that automatically validates documentation quality and catches common issues. Script should check: broken internal markdown links (scan all .md files for [text](./path.md) patterns and verify target files exist), bash code block syntax (extract all ```bash blocks and run through bash -n for syntax check), required sections present (verify TOC exists in docs over 1000 words, check for Examples heading in API.md and TROUBLESHOOTING.md), consistent formatting (heading levels don't skip, code blocks properly closed), and no dead external links if possible (basic HTTP check). Exit with non-zero code if any validation fails. Add helpful error messages showing which file and line has issues. Make script runnable in CI via .github/workflows/docs-check.yml. Target: 100-150 lines of bash. Include usage documentation in script header comments.

## Task 14: Add breadcrumb navigation to all docs

Add consistent breadcrumb navigation to the top of every documentation markdown file for easy context and navigation. Format: `[Home](README.md) > [Docs Index](DOCS.md) > Current Document Name`. Place breadcrumbs as the first line after the main title in each doc. Add to: ARCHITECTURE.md, API.md, CONTRIBUTING.md, DATABASE.md, TROUBLESHOOTING.md, DEPLOYMENT.md, TESTING.md, TYPESCRIPT.md, and expanded DEV.md. Also add "See Also" sections at the bottom of each doc with links to related documentation (e.g., CONTRIBUTING.md links to TESTING.md and ARCHITECTURE.md, API.md links to TROUBLESHOOTING.md). Create logical cross-reference graph so readers can discover related content. Estimated: 2-3 related links per doc.

## Task 15: Create architecture diagrams

Design and add ASCII or simple diagrams to illustrate system architecture and data flow. Create at minimum: request flow diagram for ARCHITECTURE.md (User → Browser → Next.js Server → API Route → Database → Response, showing each component), component hierarchy diagram showing React component nesting (AppShell → Sidebar, TopBar, main content → charts, modals), database schema diagram showing tables and relationships (if any foreign keys), and deployment architecture showing prod vs dev environments. Keep diagrams simple enough to render correctly in plain text markdown viewers. Use box-drawing characters or simple ASCII art. Include diagram source (if using tool like Mermaid, include both the Mermaid code and a fallback ASCII version). Place diagrams inline in relevant documentation sections. Estimated: 4-6 diagrams total.

## Task 16: Test all code examples

Extract and manually test every code example in all documentation files to ensure they work exactly as written. For bash commands: run each command in a fresh terminal session, verify output matches what's shown or described, check for typos in paths or command flags. For SQL queries: execute against actual database (use test data if needed), verify results are as expected, check for syntax errors. For TypeScript/JavaScript code: extract to temporary .ts files, run through compiler/linter, verify no type errors. For API curl examples: execute against running dev server, verify response structure matches documented JSON. Track tested examples in a checklist. If any example fails, fix the documentation. Goal: 100% of code examples verified working. This is the most critical quality gate - users must be able to copy-paste examples with confidence they'll work.

## Task 17: Peer review and copyedit all documentation

Conduct thorough peer review of all documentation for technical accuracy, clarity, and grammar. Have at least one other developer read each doc file and verify: technical accuracy (code examples work, architectural descriptions match actual code, API docs match implementation), clarity (explanations are understandable to target audience, no unexplained jargon, flow is logical), completeness (no major topics missing, all referenced sections exist), consistency (terminology used consistently across docs, formatting consistent, tone consistent). Use spellcheck and grammar tools on all markdown. Fix any confusing phrasing, typos, or grammatical errors. Ensure reading level is appropriate (technical but not overly academic). Review checklist should verify every item in design doc's Quality Assurance section. Aim for documentation that a junior developer or new operator can follow without clarification questions.

## Task 18: Set up CI documentation checks

Create GitHub Actions workflow (.github/workflows/docs-check.yml) that runs documentation validation on every pull request. Workflow should: checkout code, run scripts/validate-docs.sh (from Task 13), run markdown link checker (using gaurav-nelson/github-action-markdown-link-check or similar), optionally run spellcheck if feasible, post results as PR comment if validation fails, and fail the check if critical issues found (broken links, syntax errors in examples). Configure workflow to run on: pull requests that modify any .md files, manual workflow dispatch for testing. Add status badge to README.md showing docs validation status. Make validation errors actionable with clear messages about what to fix. This ensures documentation quality is maintained as codebase evolves.

## Task 19: Create onboarding test scenario

Validate documentation effectiveness by conducting a real onboarding test with a developer unfamiliar with the codebase. Recruit a developer (from team or external volunteer) who hasn't worked on financial-analyzer. Give them access to only the documentation (no live help). Ask them to: clone repo and start dev server (track time - should be under 15 min), read ARCHITECTURE.md and explain the codebase structure (should take under 30 min), implement a small feature (add a simple API endpoint or UI component) following CONTRIBUTING.md (should complete with passing tests in under 4 hours). Observe where they get stuck, what questions they ask, which docs they reference. Record feedback on: what was confusing, what was missing, what was helpful. Use this feedback to improve documentation before final merge. Goal: prove that documentation enables <1 day onboarding as specified in PRD success criteria.

## Task 20: Document the documentation update process

Add a section to CONTRIBUTING.md that explains when and how to update documentation when making code changes. Create a simple decision tree or table: "If you change X, update Y doc". For example: adding API endpoint → update API.md, changing database schema → update DATABASE.md, modifying deployment process → update DEPLOYMENT.md, changing code standards → update CONTRIBUTING.md, adding environment variable → update DEV.md. Include in PR template (if one exists) a checkbox: "Documentation updated if needed". Establish ownership: assign "documentation owner" role (engineering lead or senior dev) responsible for quarterly audits. Document the quarterly audit process: review each doc for outdated content, test code examples, check for new features not documented, review metrics (onboarding time, support questions). Add calendar reminder for January audit (annual review). This ensures documentation doesn't rot after initial creation.
